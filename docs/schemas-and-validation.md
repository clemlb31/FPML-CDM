# Schemas & structural validity

Where to provide schemas, and how to structurally validate MXML / FpML / CDM — independently of *semantic*
validation (which compares against the references in `data/`).

## Where to place schemas — [schemas/](../schemas/)

| Folder | Format | Expected content |
|---|---|---|
| [schemas/mxml/](../schemas/mxml/) | XSD | MXML XSD (provided by Murex) |
| [schemas/fpml/](../schemas/fpml/) | XSD | FpML 5.x confirmation XSD ([fpml.org](https://www.fpml.org/)) |
| [schemas/cdm/](../schemas/cdm/) | JSON Schema | (optional) CDM JSON Schema generated by rosetta |

## Validating XML (MXML, FpML) — pure JDK XSD validator

[`io.fpmlcdm.schema.XsdValidator`](../src/main/java/io/fpmlcdm/schema/XsdValidator.java) +
[`XsdValidateCli`](../src/main/java/io/fpmlcdm/schema/XsdValidateCli.java): **no CDM/Maven dependency**,
so it compiles and runs **standalone** even when the CDM build is unavailable.

```bash
# compile (once)
javac -d tmp/schema-out src/main/java/io/fpmlcdm/schema/*.java

# validate a file or a folder (exit code 1 if at least one is invalid)
java -cp tmp/schema-out io.fpmlcdm.schema.XsdValidateCli \
    --xsd schemas/fpml/fpml-main-5-3.xsd \
    --input data/ground_truth/mxml-fpml/OUT_IRD_ASWP_5-3_INS_01/OUT_IRD_ASWP_5-3_INS_01_expected.xml
```

Output: `OK`/`FAIL` per file + errors `SEVERITY [line:col] message`. The `xs:include`/`xs:import`
are resolved relative to the root XSD.

> Typical usage: validate each **MXML→FpML** output against the FpML XSD before the semantic comparison,
> and validate the input MXML against the MXML XSD.

## Validating CDM (JSON)

No XSD (it's JSON). Two levels already in place — details in [schemas/cdm/README.md](../schemas/cdm/README.md):
1. **Model** — `RosettaObjectMapper` deserialization → `TradeState` (the pipeline does this).
2. **Data-rules** — `io.fpmlcdm.validate.CdmValidator` (`noNewCdmViolations` signal).

## Build / environment (reminder)

The deps (including `cdm-java:6.19.0`) are on **Maven Central**. The default `mvn` `-takari` points to the
**Murex Nexus** (`pandora.fr.murex.com` / `repo-dev/nexus`), unreachable off VPN → resolution fails
(`No versions available …`). **Workaround: `mvn -s <settings-central.xml>`.** The XSD validator above
needs no Maven at all (pure JDK).
